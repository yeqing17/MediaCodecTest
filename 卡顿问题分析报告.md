# Android9 PAD 视频卡顿问题分析报告

## 一、问题现象

APK在Android 9 PAD设备上播放直播视频时出现：

- 视频明显卡顿
- 音视频逐渐不同步
- 怀疑MediaCodec硬解码性能不足

播放链路：

```text
HTTP播放地址
    ↓
IJKPlayer（定制版本）
    ↓
MediaCodec
    ↓
OMX.amlogic.avc.decoder.awesome
    ↓
SurfaceView
```

设备平台：

```text
Android 9
Amlogic（晶晨）平台
```

------

## 二、日志分析结果

### 1. MediaCodec初始化正常

日志显示：

```text
OMX.amlogic.avc.decoder.awesome
```

成功创建。

同时：

```text
first rendering
```

首帧成功输出。

说明：

- MediaCodec启动成功
- 硬解码器工作正常
- 不存在初始化失败

------

### 2. 未发现解码器异常

日志中未发现：

```text
CodecException
OMX Error
dequeue timeout
decoder timeout
```

等典型解码器故障日志。

因此：

目前没有证据证明MediaCodec性能不足。

------

### 3. 视频输出速度明显低于源流速度

视频流：

```text
25fps
```

对应：

```text
PTS间隔 ≈ 40ms
```

日志显示：

```text
delta pts ≈ 40ms
delta sys ≈ 56ms
```

后期甚至：

```text
delta sys ≈ 72ms
```

对应实际输出帧率：

```text
17.8fps
13.9fps
```

远低于25fps。

------

### 4. 视频积压持续增长

日志：

```text
video:es_pkt=29
video:es_pkt=54
video:es_pkt=85
video:es_pkt=108
```

持续增长。

说明：

```text
视频输入速度
>
视频消费速度
```

播放器逐渐堆积视频数据。

------

### 5. 播放器未执行丢帧追赶

日志：

```text
A-V=-0.4s
```

即：

```text
视频落后音频约400ms
```

正常播放器应主动：

```text
drop frame
```

追赶音频。

但日志显示：

```text
drop_frm=0
```

全程未发生丢帧。

------

### 6. Surface时间戳异常

系统日志持续出现：

```text
Timestamp seems implausible relative to expectedPresent
```

说明：

```text
SurfaceFlinger
BufferQueue
显示时间戳
```

存在异常。

------

## 三、当前判断

概率评估：

### 1. IJK播放器同步逻辑异常（80%）

现象：

```text
视频落后
↓
未触发丢帧
↓
视频持续积压
↓
输出帧率下降
```

与日志高度吻合。

------

### 2. Amlogic显示链路问题（15%）

现象：

```text
Surface时间戳异常
```

可能影响显示调度。

------

### 3. MediaCodec解码性能不足（5%）

目前日志证据不足。

未发现典型解码性能瓶颈特征。

------

## 四、建议方案

由于当前无法获得IJK源码。

继续分析日志价值有限。

建议采用独立播放器验证方案。

------

## 五、验证目标

验证以下问题：

### 验证1

MediaCodec是否正常工作

### 验证2

Android9系统是否存在显示问题

### 验证3

问题是否来源于IJK播放器实现

------

## 六、预期结果

### 情况A

ExoPlayer流畅播放

结论：

```text
MediaCodec正常
Android系统正常
问题位于IJK播放器链路
```

------

### 情况B

ExoPlayer同样卡顿

结论：

```text
Amlogic驱动问题
Android系统问题
Surface显示链路问题
```

------

## 七、最终建议

优先开发独立MediaCodec测试APK进行验证。

该方案成本最低、定位最快、结论最明确。
